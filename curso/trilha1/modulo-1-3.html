🔌 LLM Layer

The LLM layer is theonly piece that knowswhich provider is running. For her, you pass on messages and tools. It translates into the provider format, calls the API, translates the response into the common format, and returns. The rest of the code never knows if it's Claude, GPT or Ollama.

🎯Common interface

type Message = { role: 'system'|'user'|'assistant'|'tool', content: string, tool_calls?: ToolCall[], tool_call_id?: string };
type ToolCall = { id: string, name: string, args: Record<string, any> };
type LLMResponse = { content: string, tool_calls: ToolCall[], usage: { input: number, output: number } };

interface LLM {
  chat(messages: Message[], tools: ToolDef[]): Promise<LLMResponse>;
}

💡Why this is strategic

In 6 months, a better model will come out. In a year, two or three. Whoever has this abstraction changes it in 5 minutes. Those who don't have it, spend a week refactoring — or simply get stuck in the old model while the competition advances.

🔧 Anatomy of a tool

A tool has three parts:name(identifier),JSON Schema(description that the LLM reads to know when and how to call) andhandler(the TypeScript function that actually executes). This pattern repeats itself for any tool — Notion, Gmail, banking, scraping.

📐Example: read_file tool

export const readFile = {
  name: 'read_file',
  description: 'Lê o conteúdo de um arquivo em data/ e retorna como texto',
  schema: {
    type: 'object',
    properties: {
      path: { type: 'string', description: 'Caminho relativo dentro de data/' }
    },
    required: ['path']
  },
  handler: async ({ path }: { path: string }) => {
    const safe = ensureInside('./data', path);   // allowlist
    const content = await fs.readFile(safe, 'utf8');
    return content.slice(0, 8000);                // limita tamanho
  }
};

📊What makes a good description

  • • It sayswhen to usethe tool, not just what it does
  • • Give a short example if the usage is ambiguous
  • • Warn about limits ("returns a maximum of 8000 chars")
  • • List alternatives ("for web, use search_web; for local file, use read_file")
TOPIC 3
3

🧰 Essential tools

Don't start with 30 tools. Starts withfiveand it can cover 80% of the real tasks. Add the rest on demand, as you use the agent and notice what's missing.

🎯Minimum kit

  • read_file:read file from disk (notes, data, documents).
  • write_file:save text to file (create note, export summary).
  • search_web:search the web (Brave Search or similar — source of current facts).
  • fetch_url:read content from a specific URL (articles, docs).
  • remember_fact:update core memory (key-value).

💡Grow little by little

Each new tool is another item in the JSON Schema that goes to LLM every call. One hundred and twenty poorly written tools confuse the model. Fifteen well-designed and clearly documented he uses with precision. Quality of description matters more than quantity.

TOPIC 4
4

🔄 The agent loop

The loop is the heart of the agent. It: (1) calls LLM with messages + tools; (2) if it comestool_calls, executes each one and returns results to LLM; (3) repeat until the answer comes only in text. It is this repetition that differentiates aagentof a chatbot.

🎯Loop pseudocode

async function runAgent(userMessage: string) {
  const messages = [systemPrompt(), ...buffer(), { role: 'user', content: userMessage }];
  for (let i = 0; i < MAX_ITER; i++) {
    const res = await llm.chat(messages, tools);
    messages.push({ role: 'assistant', content: res.content, tool_calls: res.tool_calls });

    if (res.tool_calls.length === 0) {
      return res.content;                        // resposta final
    }

    for (const call of res.tool_calls) {
      const tool = tools.find(t => t.name === call.name);
      const result = await tool.handler(call.args);
      messages.push({ role: 'tool', tool_call_id: call.id, content: String(result) });
    }
  }
  throw new Error('MAX_ITER atingido sem resposta final');
}

📊Typical scenario

You ask "summarise Marcelo's last email". The loop does: (1) LLM callssearch_emails(from='marcelo'); (2) tool returns 3 emails; (3) LLM callsread_email(id=ultimo); (4) tool returns body; (5) LLM responds with the summary. There were 3 back-and-forths within one message from you.

TOPIC 5
5

🚧 Limits and security

Loop without brake is guaranteed disaster.Four protectionsare mandatory before releasing the agent: iteration limit, path allowlist, network timeout, and kill switch via env.

✓ Always do

  • • MAX_ITER of 15-20— to enter a loop
  • • Explicit Allowlistof folders where you can read/write
  • • 30s timeoutin fetch and tool calls
  • • KILL_SWITCH variablewhich turns off the agent without deploying

✗ Never do

  • • Loop without MAX_ITER (invoice explodes)
  • • Allow absolute paths in read_file
  • • Execute shell command without confirmation
  • • Trust 100% in path validated by LLM

⚠️Real loss scenario

Without MAX_ITER: agent goes into a spiral calling the same tool with small variations. You sleep. In the morning, you make $300 in a single conversation. With MAX_ITER of 15: he stops after 15 actions and asks for help. Cost: zero extra bill.

TOPIC 6

✅ Final answer on Telegram

The loop is finished, you have text. Now it needs to become a message on Telegram — and Telegram has its own rules that take everyone by surprise:limit of 4096 characters per message, annoying Markdown parser (special characters need escaping) and rate limit that punishes editing the same message too quickly.

🎯Practical recipe

  1. 1.Useparse_mode: 'HTML'instead of Markdown — escaping is more predictable.
  2. 2.If the answer exceeds 4000 characters, divide it into pieces per paragraph or block of code.
  3. 3.In a retry of 429 (rate limit), wait the time indicated by the error.
  4. 4.Saves the agent's message in the buffer before sending — if Telegram fails, you won't lose the history.
