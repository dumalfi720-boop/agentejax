🧠 Identity and Memory

Without identity, the agent responds like a generic chatbot. Without memory, he forgets everything. Here you give voice and three layers of memories that make all the difference.

TOPIC 1
1

🎭 Personality (soul.md)

A soul.mdis a Markdown file that defines who the agent is: tone, rules, response format, what he refuses to do. It is injected as a system prompt in every LLM call. Versioning as Markdown lets you edit like any other project file and see diff in git.

📋What is always in the soul

  • Identity:name of the agent, purpose, who they work for (you).
  • Tone:formal? casual? direct? good-natured? short examples of the style.
  • Rules:always respond in PT-BR, short messages on Telegram, block code, etc.
  • Refusals:what not to do (send sensitive data in plain text, destructive actions without confirmation).
  • Facts about you:name, time zone, profession, minimum family context.

💡How to evolve the soul

Start with 30 lines. Use the agent one week. Every time he gets the tone or style wrong, open the soul and add the rule ("never use emoji at the beginning of the answer", "prefer bullets over long paragraphs"). In two weeks she is sharp and the agent "sounds like you expect."

TOPIC 2
2

⚙️ Centralized configuration

Instead of spreadingprocess.env.ALGUMA_COISAthroughout the code, you createa single modulewhich reads the .env, validates types with Zod, and exports an objectconfigtyped. All the rest of the code imports from there.

🎯Why is it worth it

  • • Validation at startup:if the .env is missing or has an invalid value, it fails immediately at boot — not 3 hours later in a real request.
  • • Painless refactoring:renamed a variable? Change in one place, TypeScript points out all the references.
  • • Safe defaults: MAX_ITERdefault 15,TZdefault America/Sao_Paulo — without needing to remember.

📊Minimal anatomy

import { z } from 'zod';
import 'dotenv/config';

const Schema = z.object({
  TELEGRAM_BOT_TOKEN: z.string().min(20),
  ALLOWED_CHAT_IDS: z.string().transform(s => s.split(',').map(Number)),
  LLM_PROVIDER: z.enum(['openrouter','anthropic','openai','ollama']),
  LLM_MODEL: z.string(),
  LLM_API_KEY: z.string().min(10),
  DATABASE_PATH: z.string().default('./data/agente.db'),
  TIMEZONE: z.string().default('America/Sao_Paulo'),
  MAX_ITER: z.coerce.number().default(15),
});

export const config = Schema.parse(process.env);
export type Config = typeof config;
TOPIC 3
3

💭 Short-term memory

This is the recent history of the conversation: the latestN messagessaved in SQLite and re-sent with each LLM call. Without this, every message from you is a new conversation — he forgets what you just said.

📐How to size

  • 40-60 message buffersuits most cases — equivalent to a one-hour conversation.
  • Old messagesthey do not disappear: they remain in the bank and can be retrieved by semantic memory (topic 5).
  • When the buffer overflows, discards the oldest. In production, before discarding, summarize in a "highlight" of a line.

💡Tip

Usebetter-sqlite3instead ofsqlite3old. It's synchronous (simpler), faster, and works with TypeScript painlessly. Create index on(chat_id, created_at)— searching for the latest N is instantaneous even with 10k messages.

TOPIC 4
4

📌Fact core memory

A tablekey-valuewith central facts about you, whichalwaysenter the system prompt. Unlike the buffer (which rolls and discards), these facts are permanent until you change them.

🔑Examples of Central Facts

  • • name: "Nei"
  • • spindle: "America/Sao_Paulo"
  • • profession: "technology consultant, focus on automation"
  • • family: "wife Ana, children Pedro (8) and Mariana (5)"
  • • meta_q2: "launch agent course in June"
  • • preference_response: "short bullets, no emoji at the beginning"

✓ Good candidates

  • • Stable facts (name, zone)
  • • Long-term preferences
  • • Family/work context
  • • Quarterly goals

✗ Not here

  • • Detail of a single conversation
  • • Temporary state ("I'm at the airport")
  • • Transactional history (goes to buffer)
  • • Long documents (go to semantics)

💡Budget

Keep total core memory below~2,000 characters(~500 tokens). All of this goes into each message — if it inflates too much, it burns tokens for nothing and dilutes the agent's focus. When you pass the limit, it's time to move old facts to the historical archive.

TOPIC 5
5

🔍 Semantic memory

Every message you exchange is converted into aembedding(numeric vector that represents the meaning) and indexed in a vector database. When you ask something, the agent searches for the passages that are most similar in meaning — not in exact words.

🎯Why it changes the game

You say:"Remember that client who complained about the deadline at the beginning of the year?"

No semantic memory: he only sees the last 40 messages, doesn't find anything, replies "which one?".

With semantic memory: search for similarity (not the word "deadline"), find the February conversation with the client Marcelo, and respond with the right context.

🔧Vector Bank Options

  • Pinecone:serverless, free tier of 100k vectors, configures in 2 minutes. Good to start.
  • sqlite-vec:SQLite extension, runs locally, free of charge, without internet. Good if you already use SQLite.
  • Qdrant:open-source, runs on Docker, scalable. Good for after 100k vectors.
  • Embedding model:text-embedding-3-small (OpenAI, $0.02/M tokens) or nomic-embed via Ollama (local free).
TOPIC 6
6

🪜 3-layer strategy

The rule of thumb is simple: each layer has a cost and a benefit. Do you uselayers 1 and 2 always, andlayer 3 on demandvia tool call.

🎯Decision summary

Layer 1 — Central

It always will. ~500 tokens. Facts about you. Rare change.

Layer 2 — Recent Buffer

It always will. ~2,000-5,000 tokens. Last 40-60 messages. Continuity.

Layer 3 — Semantics

Only when the agent decides to call the toolbuscar_memoria(query). It can bring 5-10 relevant snippets.

💰Cost impact

Sending everything with each message (without this hierarchy) easily triples your monthly token spend. The difference between $5/month and $50/month comes from where you cut — and cutting smart is exactly that here.
