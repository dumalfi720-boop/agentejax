============= MODULE 1.1 =============
1.1 ~40 min

⚙️ Preparation and Environment

Everything that needs to be in place before writing the first agent line: Telegram bot, template key, Node project, protected secrets, and an end-to-end test.

Topic 1
What is it:

A bot is an automatic Telegram account created by chatting with @BotFather. In the end, you receive a token (a long string) that gives your code permission to read and send messages on the bot's behalf.

Why learn:

It is the door through which you speak to the agent. Without the bot, there is no channel — and other than that, Telegram is free, multiplatform and has a client on every device you already use.

Key concepts:

@BotFather, token, chat ID of your user (to restrict who speaks), polling vs webhook, client library (Telegraf).

Topic 2
What is it:

The credential that authorizes your code to call the AI ​​model. It can be a paid API key (Anthropic, OpenAI), a cheap aggregator (OpenRouter) or a model running locally for free (Ollama).

Why learn:

Without a model, the agent does not think. And having abstraction from the start lets you hot swap — start at DeepSeek for pennies, go to Claude when you need quality, fallback to Ollama if the internet goes down.

Key concepts:

API key, cost per token (input vs output), context window, model name, OpenRouter as "one key for everyone", Ollama to run locally at no cost.

Topic 3
What is it:

Initialize a Node.js project with TypeScript: install Node 20+, run npm init, configure tsconfig.json and have a "dev" script that runs the code.

Why learn:

It's the skeleton. TypeScript gives you types to avoid silly mistakes, hot-reload to iterate quickly and structure for the project to grow without becoming a mess.

Key concepts:

npm init, tsconfig.json (target ES2022, module Node16), tsx to run direct TS, scripts in package.json, dependency management.

Topic 4
What is it:

An .env file stores keys and configs outside of the code. .gitignore guarantees that it never goes to the public repository — and .env.example documents which variables exist without exposing values.

Why learn:

It's the difference between leaking your keys on GitHub and having the agent secure. It is the first safety rule of any serious project, and it costs zero to implement.

Key concepts:

dotenv, .env vs .env.example, .gitignore, dev/prod separation, never commit secret, secret manager on deploy (Railway, Doppler).

Topic 5
What is it:

A clear division: src/ for code, scripts/ for utilities (migrations, seeds), data/ for real-time agent files (SQLite, logs), tests/ for tests.

Why learn:

When the project grows to 50 files, having convention avoids "where do I put this?" all the time. Reduces friction and helps AI understand your code when you ask for help.

Key concepts:

Separation of responsibilities, code vs data vs config, modules per domain (memory/, tools/, llm/), index.ts as entry point.

Topic 6
What is it:

A unique script that: reads the .env token and key, connects to Telegram, calls LLM with a fixed message and responds in chat. Just to prove that everything talks.

Why learn:

It's a quick win. If this runs, you know that Telegram, LLM and your code are aligned — any feature from now on will build on this foundation.

Key concepts:

Smoke test, layered debugging (credentials → network → response), informative logs, Telegram polling, first round-trip of tokens.

============= MODULE 1.2 =============
1.2 ~45 min

🧠 Identity and Memory

Without identity, the agent responds generically. Without memory he forgets everything. Here you give him a voice, rules and three layers of memory that make him really remember you.

What is it:

A Markdown file with the agent's "system prompt": tone of voice, response format, what they do, what they refuse, and basic facts about you.

Why learn:

It's what separates a generic chatbot from an assistant that feels like yours. Tuning the soul after a week of use is where the magic happens — you teach it his style.

Key concepts:

System prompt, identity vs behavior, explicit rules (what NOT to do), Markdown as a versionable format, reload without restarting the process.

What is it:

A single module that reads the .env, validates types, exposes a typed config object (model, buffer size, time zone, limits) for the rest of the code to import.

Why learn:

Centralizing config avoids process.env spread all over the place. When you change the name of a variable, you change it in one place — and TypeScript tells you where it still uses the old one.

Key concepts:

Single source of truth, validation with Zod, typed config, default values, config vs code separation.

What is it:

The last N messages of the conversation (40, 100, whatever fits in the context), saved in SQLite and re-sent with each LLM call so it remembers what you just said.

Why learn:

It's what provides continuity — without it, each message from you is a new conversation from scratch. With that, you say "send another one like the last one" and he understands.

Key concepts:

FIFO buffer, SQLite with better-sqlite3, token count vs message count, summary when buffer overflows, role (user/assistant/system).

What is it:

A key-value table in SQLite with central facts about you: name, time zone, profession, current goals, important people. These facts go on EVERY LLM call.

Why learn:

It makes the agent never forget the essentials. You say "my wife's name is Ana" once and it remembers it forever, without having to repeat it or search the history.

Key concepts:

Key-value store, indefinite persistence, hot reload (changed the fact, next answer already uses it), size budget (~2k chars), update via tool call.

What is it:

Each message is converted into a vector (embedding) and indexed in a vector database (Pinecone, sqlite-vec). When you ask a question, look for the passages that are most similar in meaning, not in exact words.

Why learn:

It's what lets the agent remember things said months ago. You say "that customer who complained about the deadline" and he recovers the right conversation, even without using a keyword.

Key concepts:

Embedding (~1500 dimensional vector), cosine similarity, top-k retrieval, chunking, embedding models (text-embedding-3-small, voyage-2), local vs cloud option.

What is it:

The rule of thumb: central facts always go (layer 1), recent buffer always goes (layer 2), semantic search is called by tool when the agent realizes that it needs to remember something old (layer 3).

Why learn:

Sending everything with each message burns tokens for nothing. Send too little and he seems amnesiac. The right strategy is the balance between cost and memory — and it changes how much you spend per month.

Key concepts:

Cost per token, recall vs precision, when to retrieve (lazy vs eager), search tool as access to layer 3, memory hierarchy.

============= MODULE 1.3 =============
1.3 ~50 min

🛠 Brain and Tools

Where the agent goes from “just talking” to “performing a real task”. You learn the abstraction of LLM, anatomy of a tool, and the loop that brings it all together.

What is it:

A chat interface ({messages, tools}) that hides underneath is Claude, GPT, DeepSeek or Ollama. The rest of the code never knows which model is running.

Why learn:

New models come out every month. When the next one is much better, you change a string in the .env instead of rewriting half the code. It's the only abstraction that pays for itself in months.

Key concepts:

Adapter pattern, different formats (Anthropic/OpenAI/Ollama), translation of tool calls between formats, fallback chain, OpenRouter as a facilitator.

What is it:

Every tool has 3 parts: name (identifier), JSON Schema (description of parameters for LLM to know how to call), and handler (the TS function that actually executes).

Why learn:

Tools are how the agent comes out of the box. Learning the pattern once lets you add 50 new tools following the same recipe — Notion, Gmail, banking, any API.

Key concepts:

Tool definition, JSON Schema (types, required, descriptions), handler as pure function, return as string or JSON, input validation.

What is it:

The first 5 tools that every agent needs: read file, write file, search the web, read URL, and update central memory. They are the minimum kit.

Why learn:

With these 5, you can do 80% of what you want: research, notes, reminders, follow-up. The other 50 tools you'll want later follow the same pattern.

Key concepts:

Allowlist of paths (security), clear description so LLM knows when to call, small returns (large truncate), error handling as a readable string.

What is it:

The heart of the system: takes a message from the user → calls LLM → if he requested a tool, executes it and returns the result to him → repeats until he responds in plain text.

Why learn:

It's what separates "chatbot" from "agent". No loop, he just talks. With a loop, it reads files, searches for data, chains 5 actions to answer a complex question.

Key concepts:

tool_calls no response, execute handler, append do tool_result, stop condition (final response vs tool call), max iterations (15-20).

What is it:

Loop safety belt: limit of N iterations so it doesn't go into an infinite spiral, allowlist of folders where it can move, timeout on network calls.

Why learn:

Unlimited loop + bug = $300 bill in one night. Bad allowlist + confused agent = important file deleted. These are protections that cost 5 minutes to write.

Key concepts:

MAX_ITER, fail-safe, allowlist (not denylist), mandatory absolute paths, fetch timeouts, kill switch via env.

What is it:

The last piece: the loop is finished, you have the final text. Now send to Telegram with Markdown formatting (bold, code), treating long messages (cut to 4096 characters).

Why learn:

Details that seem simple and break everything in practice: Telegram does not accept standard Markdown, has a size limit, and messages with format errors silently fail.

Key concepts:

parse_mode (MarkdownV2 vs HTML), escape special characters, split long message, retry at 429 (rate limit), save the message in the buffer.
